# 鳥人間コンテスト 滑空機部門 (Team ЯTR) 電装システム・ソフトウェア開発仕様書

以下の仕様に基づき、PlatformIO (Arduinoフレームワーク / FreeRTOS環境) で動作するC++コード（ヘッダおよびソース）、ならびにAndroidアプリ用のコードを生成してください。また、Antigravity SDKを用いた構築・コンパイル・GitHubへの自動コミットを一括で行うためのPythonスクリプトも合わせて生成対象とします。

## 1. 開発環境およびシステム構成
* **IDE:** VSCode
* **フレームワーク:** PlatformIO (Arduinoフレームワークベース / FreeRTOS)
* **電源仕様:** 2S (7.4V) または 3S (11.1V) の 1000mAh LiPoバッテリー。全体消費電流を 500mA〜1A 程度と想定し、30分以上の連続稼働を担保する。
* **ターゲットMCUと構成 (各基板は独立したノードとして構成):**
  * ESP32-WROOM-32E: メイン基板(SD実装済)、ピトー管基板(SD実装済)、GPS基板、スピーカー基板
  * ESP32-WROOM-32E (CANバスから着脱式): ディスプレイ基板(ILI9341・SDスロット搭載、フライト前確認用)
  * ESP32-C3: 高度計基板
  * STM32F303K8T6: 舵角計基板

## 2. ハードウェア実装および物理的フェールセーフ仕様
* **防水・犠牲基板設計 (琵琶湖着水対策):**
  * ESP32本体やSDカードを搭載するメイン基板およびピトー管基板の中枢部は、タッパー等による完全密閉（気密・水密）構造とする。
  * 気圧計測の遅延を防ぐため、LPS22、ICM-42688、BM1422AGMVはタッパー外の「外部犠牲基板」として配置し、メイン基板からI2Cバスを延長して接続する。
* **防滴・防風処理:**
  * 外部犠牲基板は、LPS22の極小吸気孔をマスキングした上で、ハヤコート（防湿ウレタンコーティング剤）等を用いて基板全体をコーティングし、ショートを遅延させる。
  * LPS22の周囲には「連続気泡構造」のウレタンスポンジを巻き、風切り音（動圧）のノイズ防止と静圧の即時伝達を両立させる。気圧変動を遮断する独立気泡構造の使用は厳禁とする。
* **CANバス終端設計:**
  * CANバスの終端抵抗（120Ω）は、フライト前に取り外すディスプレイ基板には絶対に配置せず、機体に常時搭載されるノードに配置する。
* **STM32の有線デバッグ延長:**
  * OTA非対応のSTM32F303（舵角計）については、カウル密閉後でもファームウェアの書き換えが可能となるよう、SWDピン（GND, SWDIO, SWCLK）を延長配線し、機体外部からアクセス可能なコネクタとして露出させる。
* **STM32F303 CAN通信のハードウェア修正要件:**
  * 回路図上、CANトランシーバがPA2およびPA3に接続されているが、STM32F303K8T6の仕様上これらのピンにCAN代替機能は存在しない。PA11/PA12等へのパターンカットおよびジャンパ配線修正が行われることを前提にコードを記述する。

## 3. センサー構成・取得レートおよび通信バス設定
* **I2C通信:** Fast Mode (400 kHz)
* **UART通信:** UM982C用は 115200 bps
* **CAN通信:** 1 Mbps (トランシーバ: BD41041FJ-C)
* **各基板の役割とセンサー割り当て:**
  * **[Node 1] メイン基板:** 外部I2C犠牲基板（ICM-42688 [100Hz], BM1422AGMV [100Hz], LPS22 [75Hz]）
  * **[Node 2] ピトー管基板:** SDP32 x1 [0x21] (100Hz), SDP31 x2 [0x22, 0x23] (100Hz), BNO055 [0x28] (100Hz), SHT41 (100Hz) ※ピトー管内径4mm仕様
  * **[Node 3] 舵角計基板:** ICM-42688 (100Hz), AS5600 (100Hz)
  * **[Node 4] GPS基板:** UM982C [10Hz]
  * **[Node 5] 高度計基板:** 超音波センサー, シングルポイントLiDAR [すべて50Hz]

## 4. GPIOピンアサイン仕様
* **[Node 1] メイン基板 (ESP32-WROOM-32E):**
  * I2C (外部犠牲基板): SDA = IO21, SCL = IO22
  * CAN通信: TX = IO32, RX = IO33
  * SPI (SDカード): SCK = IO18, MISO = IO19, MOSI = IO23, CS = IO4
  * バッテリー電圧監視 (ADC): IO34 または IO35（確定待ち）
* **[Node 2] ピトー管基板 (ESP32-WROOM-32E):**
  * I2C (センサー): SDA = IO21, SCL = IO22
  * CAN通信: TX = IO32, RX = IO33
  * SPI (SDカード): SCK = IO18, MISO = IO19, MOSI = IO23, CS = IO4
* **[Node 3] 舵角計基板 (STM32F303K8T6):**
  * I2C (センサー): SDA = PA14, SCL = PA15 (アドレス: `0x68`)
  * CAN通信: TX = PA12, RX = PA11 (※回路図のPA2/PA3から物理修正前提)
  * 外部割り込み: INT1 = PA0, INT2 = PA1
  * 制御用: LED = PA4, UART/Debug = PA9, PA10
* **[Node 6] スピーカー基板 (ESP32-WROOM-32E):**
  * UART通信 (音声モジュールへ): TX = IO32, RX = IO33
  * I2C: SDA = IO21, SCL = IO22
* **[Node 7] ディスプレイ基板 (ESP32-WROOM-32E):**
  * SPI (ILI9341): SCK = IO18, MISO = IO19, MOSI = IO23
  * ディスプレイ制御: CS = IO5, DC = IO17, RST = IO16
  * CAN通信: TX = IO25, RX = IO26
  * SPI (SDカード): SCK = IO14, MISO = IO33, MOSI = IO13, CS = IO15
  * I2C (RTC等): SDA = IO21, SCL = IO22
  * スイッチ入力: IO12

## 5. マルチスレッド（FreeRTOS）アーキテクチャ
* **タスク優先順位とコア割り当て:**
  * **Core 0 (PRO_CPU):** SDカード書き込み(5Hz) [優先度: 中]、Bluetooth通信 [優先度: 低]、タスク・ウォッチドッグ(TWDT)
  * **Core 1 (APP_CPU):** センサー取得 [優先度: 高]、CAN通信(TX/RX) [優先度: 中]、音声出力制御、ADC監視
* **タスク間通信 (IPC):** `StreamBuffer` またはLock-freeリングバッファを使用し、高優先度タスクのブロックを回避する。

## 6. データ処理・堅牢性仕様
* **内部演算フォーマット:** すべて `float` で統一。
* **絶対時刻同期とPPS連携:** UM982CのNMEAデータとPPSピン割り込みでRTCを校正し、ログのタイムスタンプに使用。
* **バッテリー電圧監視:** ESP32のADCピンを使用し抵抗分圧された値を実電圧へ変換。
* **耐障害性 (I2Cリカバリ):** 着水による外部基板のI2Cショート等で通信タイムアウトが発生した場合、SCLピンをGPIOとしてダミークロック（トグル）出力し、スレーブのロック状態を強制解除するリカバリ機構を実装。無限待機は厳禁。
* **動的キャリブレーション:**
  * Bluetooth経由で `>CMD:CALIB_ZERO\n` 受信時、直後から3秒間のデータをバッファリングし、外れ値を除去（四分位範囲等を利用）した平均値を算出し、差圧センサー（SDP31およびSDP32）の基準値（ゼロ点）として適用する。

## 7. SDカード保存仕様 (SPI接続)
* **書き込み手法:** ピンポンバッファ（ダブルバッファリング）およびESP32内蔵SPIコントローラのDMA転送を使用し、SDカード特有のページ消去遅延によるセンサー取得タスクのジッタを完全に排除する。
* **書き込み頻度:** 5Hz（200ms）間隔でのバッチ書き込み。
* **ファイル連番命名規則:** 起動時・再マウント時に連番（例: `log_0001.bin`）を付与して常に新規作成。
* **データ形式:** 固定長バイナリフォーマット。
  [Sync 0xAA55(2B)] + [Timestamp(8B)] + [Sensor ID(1B)] + [Length(1B)] + [Payload(NB)] + [CRC16(2B)]

### 8-2. フライトシミュレータ連携 (UDP-CAN ブリッジ)
* **対象ファイル:** `src/udp_can_bridge.cpp` を新規作成。
* **処理フロー:** シミュレータからのUDPパケット（Port: 8888）を受信し、ヘッダ検証（`[0]==255`, `[1]==255`, `[7]==255`）を通過した高度データ（`float`）を抽出する。
* **データ転送:** 抽出した高度データを、音声フィードバックのソースデータとして使用するため、CAN ID **`0x042`** (超音波センサー) としてCANバス（1Mbps）へ送信する。
* **エラーハンドリング:** 起動時に3秒間のディレイを設け、WiFi/UDP/CANのいずれかの初期化に失敗した場合は無限ループへ移行し、エラーログをシリアル出力する。
* **不要処理の排除:** 旧システムで使用されていた膨大なSwitch文によるパケット振り分けやデバッグログ出力はすべて排除し、処理を最適化する。

### 8-3. 有機EL/液晶ディスプレイ基板
* `Adafruit_ILI9341` および `Adafruit_GFX` を使用。ハードウェアSPIを利用し約15Hzの描画更新レートを確保する。
* **UI表示設計:** 物理ボタンによるページ切り替えは行わず、1つの画面内にすべての必須情報を一覧表示できる視認性の高いレイアウトを構築すること。
* **欠損データのハンドリング:** CANバスからまだ受信していないデータ（未達データ）については、初期値の `0` ではなく `NULL` や `No Data` 等の明確な文字列を表示し、データの欠落を即座に判別できるようにすること。

### 8-4. スピーカー基板
* AquesTalk Pico LSIを使用。UART通信（TX/RX）を介してテキストストリームを受信し、日本語での音声合成・発声を行うモジュール制御を実装する。

### 8-5. Androidアプリケーション (テレメトリUI)
* **開発言語:** Kotlin
* Bluetooth SPP経由でテキストを受信し、グラフと数値で描画。動的キャリブレーションおよびOTA開始コマンド送信ボタンを実装する。

### 8-6. OTA (Over-The-Air) ファームウェア更新
* Androidアプリからの `>CMD:OTA_START\n` 受信時のみWi-Fi機能を起動し、無線アップデートを待機する。アップデート完了またはタイムアウトで再起動し、通常運用時（Wi-Fiオフ）のバッテリー消費を抑える。

## 9. AIコード生成ルール
* **既存ライブラリの排他制御:** センサー読み取りに既存ライブラリを使用する際、FreeRTOSでのI2Cクラッシュを防ぐため、ライブラリの呼び出し前後を `xSemaphoreTake` および `xSemaphoreGive` を用いたミューテックスで囲むラッパークラスを実装すること。
* **ヘッダ駆動開発:** 先に `.hpp` ファイルでDoxygen形式のコメントを用いてクラスや関数の仕様を厳密に定義すること。
* **生成場所の通知:** コード生成完了後、マイコンへ書き込む対象のコードがどのディレクトリ・ファイル名で作成されたかを、ユーザーに対して明確に提示すること。
