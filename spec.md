# 鳥人間コンテスト 滑空機部門 (Team ЯTR) 電装システム・ソフトウェア開発仕様書

以下の仕様に基づき、PlatformIO (Arduinoフレームワーク / FreeRTOS環境) で動作するC++コード（ヘッダおよびソース）、ならびにAndroidアプリ用のコードを生成してください。

## 1. 開発環境およびシステム構成
* **IDE:** VSCode
* **フレームワーク:** PlatformIO (Arduinoフレームワークベース / FreeRTOS)
* **電源仕様:** 2S (7.4V) または 3S (11.1V) の 1000mAh LiPoバッテリー。全体消費電流を 500mA〜1A 程度と想定し、30分以上の連続稼働を担保する。
* **ターゲットMCUと構成 (各基板は独立したノードとして構成):**
  * ESP32-WROOM-32E: メイン基板(SD実装済)、ピトー管基板(SD実装済)、GPS基板、スピーカー基板
  * ESP32-WROOM-32E (CANバスから着脱式): ディスプレイ基板(ILI9341搭載、フライト前確認用)
  * ESP32-C3: 高度計基板
  * STM32F303: 舵角計基板

## 2. ハードウェア実装および物理的フェールセーフ仕様
* **防水・犠牲基板設計:**
  * ESP32本体やSDカードを搭載するメイン基板は、タッパー等による完全密閉（気密・水密）構造とする。
  * 気圧計測の遅延を防ぐため、LPS22、ICM-42688、BM1422AGMVはタッパー外の「外部犠牲基板」として配置し、メイン基板からI2Cバスを延長して接続する。
* **防滴・防風処理:**
  * 外部犠牲基板は、LPS22の吸気孔をマスキングした上で、ハヤコート（防湿ウレタンコーティング剤）等を用いて基板全体をコーティングする。
  * LPS22の周囲には連続気泡構造のウレタンスポンジを巻き、風切り音（動圧）のノイズ防止と気圧の即時伝達を両立させる。独立気泡構造（気密性のあるスポンジ）の使用は厳禁とする。
* **CANバス終端設計:**
  * CANバスの終端抵抗（120Ω）は、フライト前に取り外すディスプレイ基板には絶対に配置せず、機体に常時搭載されるノード（メイン基板等）に配置する。
* **STM32の有線デバッグ延長:**
  * OTA非対応のSTM32F303については、カウル密閉後でもファームウェアの書き換えが可能となるよう、SWDピン（GND, SWDIO, SWCLK）を延長配線し、機体外部からアクセス可能なコネクタとして露出させる。

## 3. センサー構成・取得レートおよび通信バス設定
* **I2C通信:** Fast Mode (400 kHz)
* **UART通信:** UM982C用は 115200 bps
* **CAN通信:** 1 Mbps (トランシーバ: BD41041FJ-C)
* **各基板のセンサー割り当て:**
  * **ピトー管基板:** SDP32 x1 (0x21), SDP31 x2 (0x22, 0x23), BNO055 (0x28), SHT41 [すべて100Hz] ※ピトー管内径4mm仕様
  * **メイン基板 (外部犠牲I2Cバス):** ICM-42688 (100Hz), BM1422AGMV (100Hz), LPS22 (75Hz)
  * **舵角計基板:** ICM-42688 (100Hz), AS5600 (100Hz)
  * **GPS基板:** UM982C [10Hz]
  * **高度計基板:** 超音波センサー, シングルポイントLiDAR [すべて50Hz]

## 4. マルチスレッド（FreeRTOS）アーキテクチャ
* **タスク優先順位設計:**
  * **[高]** センサーデータ取得タスク
  * **[中]** CAN通信タスク (TX/RX)、SDカード書き込みタスク
  * **[低]** Bluetooth SPP テレメトリ通信タスク
* **Core 0 (PRO_CPU):** SDカード書き込み(5Hz)、Bluetooth通信、タスク・ウォッチドッグ(TWDT)
* **Core 1 (APP_CPU):** センサー取得、CAN通信、音声出力制御、バッテリー電圧監視
* **タスク間通信 (IPC):** `StreamBuffer` またはLock-freeリングバッファを使用。

## 5. データ処理・堅牢性仕様
* **内部演算フォーマット:** すべて `float` で統一。
* **絶対時刻同期とPPS連携:** UM982CのNMEAデータとPPSピン割り込みでRTCを校正し、ログのタイムスタンプに使用。
* **バッテリー電圧監視:** ESP32のADCピンを使用し抵抗分圧された値を実電圧へ変換。
* **耐障害性 (I2Cリカバリ):** 着水による外部基板のI2Cショート等で通信タイムアウトが発生した場合、SCLピンをGPIOとしてダミークロック（トグル）出力し、スレーブのロック状態を強制解除するリカバリ機構を実装。無限待機は厳禁。
* **動的キャリブレーション:**
  * `>CMD:CALIB_ZERO\n` 受信時、直後から3秒間のデータをバッファリングし、外れ値を除去（四分位範囲等を利用）した平均値を算出し、差圧センサー（SDP31およびSDP32）の基準値（ゼロ点）として適用する。

## 6. SDカード保存仕様 (SPI接続)
* **書き込み手法:** ピンポンバッファ（ダブルバッファリング）およびESP32内蔵SPIコントローラのDMA（Direct Memory Access）転送を使用し、SDカード特Requirements有の遅延によるセンサー取得タスクのブロックを完全に排除する。
* **書き込み頻度:** 5Hz（200ms）間隔でのバッチ書き込み。
* **ファイル連番命名規則:** 起動時・再マウント時に連番（例: `log_0001.bin`）を付与して常に新規作成。
* **データ形式:** 固定長バイナリフォーマット。
  [Sync 0xAA55(2B)] + [Timestamp(8B)] + [Sensor ID(1B)] + [Length(1B)] + [Payload(NB)] + [CRC16(2B)]

## 7. 通信およびUI仕様
### 7-1. CAN通信
* 内部の `float` 値を静的定義された各センサーの分解能スケールで乗算し、整数キャストして送信。送受信間で共通のスケーリング定数ヘッダを作成すること。
### 7-2. 有機EL/液晶ディスプレイ基板 (CANバスから着脱式独立ノード / ILI9341搭載)
* フライト前の確認用ノードとしてCANバスに参加しデータを受信。離陸前の物理的な取り外しを前提とする。
* `Adafruit_ILI9341` および `Adafruit_GFX` を使用。ハードウェアSPIを利用し約15Hzの描画更新レートを確保。物理ボタンによるページ切り替えUIを実装。
### 7-3. スピーカー基板 (独立ESP32ノード)
* UART通信等を介してテキストストリームを受信し、日本語での音声合成・発声を行うモジュール制御を実装。
### 7-4. Androidアプリケーション (テレメトリUI)
* **開発言語:** Kotlin
* Bluetooth SPP経由でテキストを受信し、グラフと数値で描画。動的キャリブレーションおよびOTA開始コマンド送信ボタンを実装。
### 7-5. OTA (Over-The-Air) ファームウェア更新 (ESP32系ノード限定)
* Androidアプリからの `>CMD:OTA_START\n` 受信時のみWi-Fi機能を起動し、無線アップデートを待機する。アップデート完了またはタイムアウトで再起動し、通常運用時（Wi-Fiオフ）のバッテリー消費を抑える。

## 8. AIコード生成ルール
* **既存ライブラリの排他制御:** センサー読み取りに既存ライブラリを使用する際、FreeRTOSでのI2Cクラッシュを防ぐため、ライブラリの呼び出し前後を `xSemaphoreTake` および `xSemaphoreGive` を用いたミューテックスで囲むラッパークラスを実装すること。
* **ヘッダ駆動開発:** 先に `.hpp` ファイルでDoxygen形式のコメントを用いてクラスや関数の仕様を厳密に定義すること。
